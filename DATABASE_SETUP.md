# Database Setup Guide for AI Study Assistant

## Database Schema

### Table 1: `chats`
Create this table in Supabase with the following structure:

```sql
CREATE TABLE chats (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  chat_id TEXT NOT NULL UNIQUE,
  chat_title TEXT NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

**Fields:**
- `id`: Auto-generated primary key
- `chat_id`: Unique identifier for each conversation (sent from frontend)
- `chat_title`: Title of the chat conversation
- `created_at`: Timestamp when chat was created

---

### Table 2: `messages`
Create this table in Supabase with the following structure:

```sql
CREATE TABLE messages (
  id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
  chat_id TEXT NOT NULL REFERENCES chats(chat_id) ON DELETE CASCADE,
  user_message TEXT NOT NULL,
  ai_reply TEXT NOT NULL,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

**Fields:**
- `id`: Auto-generated primary key
- `chat_id`: Foreign key reference to `chats.chat_id`
- `user_message`: The message sent by the user
- `ai_reply`: The AI response
- `created_at`: Timestamp when message was created

**Important:** The `ON DELETE CASCADE` ensures that when a chat is deleted, all related messages are automatically deleted.

---

## Steps to Set Up in Supabase

1. **Go to SQL Editor** in your Supabase dashboard
2. **Copy and run the SQL above** to create both tables
3. **Verify** that both tables exist in your database
4. **Enable Row Level Security (RLS)** if needed for security

---

## Architecture Overview

### How It Works:

1. **Chat Creation**: When a new chat starts, ONE row is created in the `chats` table
2. **Message Saving**: Every user/AI message pair is saved to the `messages` table
3. **Chat Listing**: The sidebar fetches from `chats` table only (no duplicates)
4. **Message Loading**: When selecting a chat, messages are fetched from `messages` table
5. **Chat Deletion**: Deletes from both tables (with cascade delete)
6. **Chat Editing**: Updates only the `chat_title` in `chats` table

## API Routes

### GET /api/chats
- **Purpose**: Fetch all unique chats for the sidebar
- **Response**: Array of chats with `chat_id`, `chat_title`, `created_at`
- **Query**: `SELECT * FROM chats ORDER BY created_at DESC`

### GET /api/messages?chatId={chatId}
- **Purpose**: Fetch all messages for a specific chat
- **Response**: Array of messages with `user_message`, `ai_reply`, `created_at`
- **Query**: `SELECT * FROM messages WHERE chat_id = {chatId} ORDER BY created_at ASC`

### POST /api/chat
- **Purpose**: Send a message and get AI response
- **Logic**:
  1. Check if `chat_id` exists in `chats` table
  2. If not, create ONE row in `chats` table
  3. Save user + AI message pair to `messages` table

### PATCH /api/chats/update
- **Purpose**: Update chat title
- **Query**: `UPDATE chats SET chat_title = {title} WHERE chat_id = {id}`

### DELETE /api/chats/delete?chatId={chatId}
- **Purpose**: Delete chat and all messages
- **Logic**:
  1. Delete from `messages` table (where chat_id = {chatId})
  2. Delete from `chats` table (where chat_id = {chatId})
  3. Cascade delete handles orphaned messages

## Troubleshooting

### Problem: Still seeing duplicate chats?
**Solution**: 
- Delete all rows from `messages` and `chats` tables
- Restart the app
- Send a new message - should create only ONE chat row

### Problem: Delete not working?
**Solution**:
- Ensure both tables exist
- Check that `messages` table has `ON DELETE CASCADE`
- Verify chatId parameter is correct

### Problem: Messages not loading?
**Solution**:
- Verify `messages` table exists
- Check that messages are being saved with correct `chat_id`
- Use Supabase dashboard to manually verify data
